# Plan for corepost-preboot

## Роль репозитория

Предзагрузочный модуль, который поднимает сеть, проверяет состояние устройства на сервере и участвует в 2FA/3FA расшифровке LUKS до загрузки ОС.

## Текущее состояние

- Текущий код завязан на `mkinitcpio` и фактически на Arch-style workflow, а не на чистый Debian.
- В старом `hook` были зашиты конкретные устройства, интерфейсы и адреса.
- Нет полноценного 3FA, нет внешней конфигурации, нет нормальной обработки ошибок и нет контракта с install repo.
- Подпись формируется от `device_id`, а не от настоящего секрета.

## Что сделать

- Определить целевую Debian-совместимую реализацию:
  - либо `initramfs-tools`;
  - либо `dracut`, если это упростит интеграцию и тесты в QEMU.
- Разделить preboot runtime и install/provisioning: этот repo отвечает только за ранний boot flow.
- Вынести все параметры в конфиг, который будет генерировать `corepost-install`:
  - server URL;
  - device id;
  - секрет HMAC;
  - профиль 2FA/3FA;
  - сетевой режим;
  - политика при потере сети.
- Реализовать реальные сценарии:
  - штатный unlock 2FA;
  - unlock 3FA с USB-фактором;
  - deny при panic-lock;
  - отказ по неверной подписи/просроченному окну;
  - поведение при отсутствии сети;
  - профиль нестабильного Wi-Fi/повторных попыток.
- Подготовить упаковку бинарников и зависимостей для initramfs: `cryptsetup`, `curl`, `openssl`, сертификаты CA, сетевые утилиты.
- Добавить тестовый стенд в QEMU на очень чистом Debian:
  - генерация диска;
  - шифрование LUKS;
  - сборка initramfs;
  - сценарии boot/unlock/fail.
- Задокументировать, как проверяется наличие всего тулчейна внутри VM и как воспроизводится запуск на macOS-host через QEMU.

## Критерий готовности

- На Debian VM в QEMU воспроизводятся штатные и негативные сценарии 2FA/3FA, lock-deny и network-deny.
- Конфигурация preboot не содержит хардкода устройства, сети и серверного адреса.
